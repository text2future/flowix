# 加载性能问题清单（按影响排序）

> 记录 Flowix 编辑器在加载路径上仍存在的几类性能瓶颈，按 P0/P1/P2 分级，并给出可选优化方向与下一步建议。

## P0 / P1 —— 仍在首屏路径上

### 1. Markdown -> ProseMirror 解析与 setContent 同步主线程
- 位置：`D:/02 vibecoding/test/flowix-main/app/flowix-web/components/editor/markdown-editor.tsx`
- 现象：初始化和切换内容时都会 `setContent(..., contentType: 'markdown')`。文档越大、节点越多，主线程卡顿越明显。
- 当前评估：**剩余最大瓶颈**。
- 可选方向（改动都比较大）：
  - 预解析 / 分片加载
  - 先渲染只读 preview，再挂编辑器
  - Worker 解析

### 2. 代码高亮 Shiki 在加载路径上偏重
- 位置：`D:/02 vibecoding/test/flowix-main/app/flowix-web/components/editor/extensions/codeblock-shiki/`
- 现象：`CodeBlockShiki` 模块 import 时会立即 `loadHighlighter(...)`，预加载主题和多种语言。代码块多、语言多时，decorations 与高亮仍可能卡。
- 可优化：
  - 高亮按可视区域 / `requestIdleCallback` / 按语言懒加载
  - 首屏只显示 plaintext，后续渐进高亮

### 3. 每个代码块同步创建完整语言下拉列表
- 位置：`D:/02 vibecoding/test/flowix-main/app/flowix-web/components/editor/extensions/codeblock-shiki/codeblock-shiki-view.ts`
- 现象：`createLanguageDropdown()` 会遍历 `bundledLanguagesInfo`，为每个代码块创建一整套 button。文档里代码块多时，会在 editor mount / setContent 期间同步放大 DOM。
- 可优化：
  - 点击语言按钮时再创建 dropdown
  - 共享一个浮层

## P1 / P2 —— 可优化

### 4. AgentThreadCard 挂载时主动 loadThreadCache
- 位置：`D:/02 vibecoding/test/flowix-main/app/flowix-web/components/editor/extensions/agent-thread-card.tsx`
- 现象：每个卡片创建时如果有 `threadId` 就触发缓存加载。文档里有多个 agent 卡片时，会和文档切换抢 IPC / 渲染资源。
- 可优化：
  - 可视区域内再加载
  - 折叠态 / 打开面板时再加载

### 5. AgentThreadCard 每次 chat store 更新会重渲染全部消息 DOM
- 现象：订阅整个 chat store 后 `renderThreadState()`，内部 `body.replaceChildren()` 全量重建消息。加载大历史或后台流式消息时比较重。
- 可优化：
  - 只订阅当前 thread
  - 做消息增量 patch
  - 限制卡片内首屏消息数量

### 6. 切换文档仍然等待 outgoing flush
- 现象：侧边栏选中态已经解耦，但真正文档 session commit 仍要等 `flushDocumentPath()`。这是数据一致性策略，保守但会拖慢“新文档内容出现”。
- 可优化（要谨慎）：
  - 后台 flush + buffer 快照隔离
  - 失败提示 / 冲突恢复
- 评估：能做但比当前实现风险高。

### 7. 外部变更 reload 仍会整文档重读 + 整体 setContent
- 现象：文件 watcher 已有 dedup，但 reload 后还是整篇 `readDocument` + `setContent`。大文档外部变更时仍会卡。
- 可优化：
  - 差异更新
  - 只在当前文档空闲时应用外部内容

### 8. 表格、任务列表、NoteReference、MentionNote 等扩展统一在 schema 初始化中加载
- 现象：编辑器每次挂载都带全套扩展。功能完整，但初始化成本固定偏高。
- 可优化：
  - 部分 UI / 菜单懒挂
  - 只在交互时需要的浮层组件延迟渲染

## 下一步建议（低风险优先）

优先做以下两项，风险小于 Worker 化 markdown 解析，收益也较直接：

1. **代码块语言下拉延迟创建**：把 `createLanguageDropdown()` 改为按需（点击按钮或可见时再构建 / 共享浮层）。
2. **AgentThreadCard 的 thread cache 改为可视 / 展开后加载**：使用 IntersectionObserver 或折叠态触发，避开首屏和文档切换期的资源竞争。

后续可考虑：

- Shiki 懒加载（按语言 / idle / 可视区域）
- AgentThreadCard 细粒度订阅 + 消息增量 patch
- 切换文档 outgoing flush 后台化（高风险，需配套冲突恢复）
