# 多 Agent 并行 Thread Card 流式展示 PRD

## 一、项目目标

建立一套统一的 Agent Thread Card 展示机制，使 Flowix 前端能够同时运行多个 Agent，并在同一编辑器环境中稳定展示不同 AgentType 的流式消息、工具调用、运行状态和历史记录。

该能力的目标不是单独接入 Codex，而是让 Codex、Flowix、Claude 等 Agent 都可以复用同一种会话卡片体验，降低后续新增 AgentType 的接入成本。

## 二、产品定位

Thread Card 是笔记中的 Agent 会话入口，而不是消息正文的存储容器。

笔记正文只保存会话锚点，包括 threadId、agentType、标题、折叠状态等基础信息。实时消息、工具调用、运行状态和错误状态由运行时状态层承接，历史消息由独立缓存恢复。

## 三、核心用户场景

用户在编辑笔记时，可以同时发起多个 Agent 任务，例如 Codex 处理代码方案、Flowix 总结笔记内容、另一个 Agent 做资料分析。每个任务都以独立 Thread Card 的形式挂载在当前笔记中，并在执行过程中实时展示进度。

用户切换笔记、折叠卡片、滚动页面或重新打开应用时，Thread Card 仍能恢复对应会话内容，并区分不同 AgentType 的消息与工具记录。

## 四、产品原则

- 统一体验：不同 AgentType 在前端表现为统一的消息流、状态流和工具调用流。
- 运行隔离：每个 Agent Run 都有独立生命周期，避免并行任务之间出现消息串流、状态覆盖或取消误伤。
- 轻量笔记：Markdown 中只保存会话入口，不将流式正文持续写回笔记，避免影响编辑器性能和光标稳定性。
- 按需加载：历史缓存只在卡片可见、展开或被用户主动访问时加载，避免多卡片文档初始化时抢占资源。
- 渐进渲染：流式内容进入前端后按 thread 维度合并刷新，保证多个 Agent 并行时仍能保持编辑器流畅。

## 五、信息架构

Agent 会话由三层对象组成：

- AgentType：表示 Agent 来源与能力类型，例如 codex、flowix、claude。
- Thread：表示一个长期会话，是 Thread Card 的主要绑定对象。
- Run：表示一次具体执行，同一个 Thread 下可以有多次 Run，也可以存在并行或续跑。

前端内部以 agentType、threadId、runId 共同识别一次运行实例，以 agentType、threadId 识别一张 Thread Card 的会话状态。

## 六、关键能力范围

- 支持多个 Agent 并行运行。
- 支持不同 AgentType 统一接入。
- 支持流式展示 assistant 消息。
- 支持工具调用状态展示。
- 支持运行中、完成、失败、取消等状态。
- 支持 Thread Card 折叠、展开、可视区域懒加载。
- 支持历史消息缓存恢复。
- 支持单个 Agent Run 独立取消。
- 支持未来扩展子 Agent、父子任务和多 Agent 协作视图。

## 七、非目标范围

- 不在 Markdown 中保存完整消息正文。
- 不要求所有 Agent 使用相同后端实现。
- 不在第一阶段实现复杂多 Agent 编排视图。
- 不在第一阶段重做编辑器整体渲染架构。
- 不把 Codex 的原始事件格式直接暴露给前端 UI。

## 八、前端体验设计

Thread Card 默认展示 Agent 类型、任务标题、运行状态、最近一条消息和主要操作。

展开后展示完整消息流，包括用户消息、Agent 回复、工具调用、错误提示和完成状态。

运行中消息以流式方式持续增长，用户可以在输出过程中继续编辑笔记。

当多个 Thread Card 同时运行时，各卡片只更新自身内容，不触发其他卡片重渲染。

当卡片折叠时，保留状态提示和必要进度，不渲染完整消息 DOM。

## 九、AgentType 兼容策略

每种 AgentType 通过独立 Adapter 接入，Adapter 负责启动任务、解析原始流、标准化事件和处理取消。

前端 Store 和 Thread Card 只消费统一事件协议，不直接依赖某个 Agent 的原始输出格式。

不同 Agent 的特殊工具调用、权限提示或文件操作展示，通过 Agent Registry 扩展，而不是写死在 Thread Card 主逻辑中。

## 十、状态管理策略

前端维护按 Thread 隔离的状态树。每个 Thread 只通知订阅自己的 Thread Card。

每个 Run 有独立状态，包括排队中、运行中、完成、失败、取消。

消息以 messageId 维度更新，流式内容只追加到对应消息，不重建整张卡片。

高频流式更新按帧合并，避免每个 token 都触发 DOM 更新。

## 十一、缓存与恢复策略

实时运行依赖内存状态，历史恢复依赖缓存。

缓存按 AgentType 和 ThreadId 隔离，避免不同 Agent 会话冲突。

Thread Card 首屏不主动加载所有历史缓存，只有在卡片可见、展开或用户进入会话时再加载。

缓存恢复不能覆盖正在运行中的实时消息，需要进行状态合并。

## 十二、异常与边界场景

Agent 启动失败时，Thread Card 保留任务入口并展示失败状态。

流式连接中断时，保留已接收消息，并允许用户重试或继续。

用户取消某个 Run 时，只影响当前 Run，不影响同文档其他 Agent。

同一 Thread 多次运行时，历史消息与新 Run 状态需要清晰区分。

缓存读取失败时，不影响实时运行，只提示历史恢复不可用。

## 十三、验收标准

- 同一笔记中可以同时运行两个以上 Agent，消息不会串流。
- Codex 和 Flowix 可以通过同一 Thread Card UI 展示基础消息流。
- 运行中的 Agent 消息能够实时增量展示。
- 折叠卡片不会持续重渲染完整消息列表。
- 多个 Agent 同时输出时，编辑器仍可正常输入和滚动。
- 重新打开包含 Thread Card 的笔记后，历史会话可以按需恢复。
- 取消一个 Agent Run 不影响其他正在运行的 Agent。
- 新增一个 AgentType 时，不需要改动 Thread Card 的核心渲染模型。

## 十四、实施阶段

第一阶段：建立统一 AgentEvent 协议和 per-thread store，完成 Codex 与现有 Flowix Agent 的基础兼容。

第二阶段：优化 Thread Card 渲染路径，支持按可视区域加载缓存、按 thread 维度批量刷新和折叠态轻量展示。

第三阶段：完善 Agent Registry，抽象不同 AgentType 的工具调用展示、能力声明和错误状态。

第四阶段：补齐缓存恢复、取消、重试、失败恢复和多 Run 历史展示。

第五阶段：为多 Agent 协作、子 Agent、父子任务关系预留扩展视图。

## 十五、成功指标

- 多 Agent 并行运行时消息串流率为 0。
- Thread Card 更新不引发整篇文档明显卡顿。
- 新增 AgentType 的接入主要集中在 Adapter 和 Registry，不侵入核心 UI。
- 历史缓存加载不阻塞文档首屏打开。
- 用户能够清楚识别每张卡片所属 Agent、当前状态和执行结果。
