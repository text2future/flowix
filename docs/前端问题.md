# 前端与桌面端剩余问题清单

> 更新时间：2026-06-18
>
> 范围：基于本次后端架构审查与后续修复后的剩余问题。已处理的 P1-1 锁中毒恢复、P1-2 启动期初始化降级、P1-3 中的 LLM 请求总超时与 429/5xx 初始重试，不再作为待办列入。P1-3 中尚未完成的 SSE 字节级解析仍保留。

## P0 / 上线阻断

### P0-1. API Key 明文存储并完整返回前端

位置：
- `app/flowix-desktop/src/user_config.rs`
- `app/flowix-desktop/src/commands/settings.rs`
- `app/flowix-web/windows/main/agent-panel/agent-root.tsx`

问题：
- `~/.flowix/ai_config.json` 仍保存明文 `api_key`。
- `get_ai_config` 仍会把完整 `AiConfigFile` 返回 renderer。
- Windows 下文件权限加固函数是 no-op，不能依赖本地 JSON 权限保护。

建议：
- API key 迁移到 OS keychain / Stronghold。
- IPC 只返回 `provider/model/apiUrl/hasKey`，密钥不经过前端和 Zustand。
- 前端保存配置时只提交新密钥或清除密钥动作，不读回密钥原文。

### P0-2. `write_export_file` 后端允许任意路径写入

位置：
- `app/flowix-desktop/src/commands/dialog.rs`

问题：
- `write_export_file(file_path, content)` 直接写入前端传入路径。
- 当前依赖 caller 侧弱校验；一旦出现 XSS、恶意依赖或调试脚本，可覆盖用户任意文件。

建议：
- 后端记录最近一次 native save dialog 返回的路径 token。
- `write_export_file` 只允许写入该 token 对应目录/文件。
- 增加扩展名、大小、覆盖策略限制。

### P0-3. CSP 与 assetProtocol 过宽

位置：
- `app/flowix-desktop/tauri.conf.json`

问题：
- `script-src 'unsafe-eval'`、`style-src 'unsafe-inline'`、`connect-src https:` 范围过大。
- `assetProtocol.scope: ["**"]` 等于把整个磁盘暴露给 asset protocol 读取范围。

建议：
- 生产环境去掉 `unsafe-eval`。
- `connect-src` 收紧到明确的 LLM provider 域名。
- `assetProtocol.scope` 收紧到 notebook 根目录与 attachments 目录。

### P0-4. AI Agent 文件访问授权边界仍偏宽

位置：
- `app/flowix-desktop/src/agent_access.rs`
- `app/flowix-desktop/src/providers/tools/mod.rs`
- `app/flowix-desktop/src/providers/tools/filesystem.rs`

问题：
- `AgentAccessKind::Folder` 允许用户把任意本地目录加入 AI 可访问范围。
- 当前 read/write 权限未拆分，LLM 一旦拿到目录授权即可读写该目录下文件。

建议：
- 默认只允许 notebook 根目录和 `attachments/`。
- 自定义 folder 拆成 read/write 权限，write 默认关闭。
- 增加危险目录 denylist，例如 `.ssh`、云密钥目录、shell profile、系统目录。
- `create_dirs` 默认关闭，symlink 做逐段检查。

### P0-5. Codex CLI 可由 UI payload 透传到 `danger-full-access`

位置：
- `app/flowix-desktop/src/codex_cli.rs`

问题：
- `permission_mode` 支持 `danger-full-access` 并透传给 Codex CLI。
- cwd 也来自消息 payload 的 `system_reminder_directory`。

建议：
- 默认仅允许 `read-only` 或 `workspace-write`。
- `danger-full-access` 必须走本机显式确认，不应由普通消息 payload 决定。
- 对 cwd 做已授权 workspace 校验。

## P1 / 高优先级

### P1-1. SSE 解析仍按字符串切包

位置：
- `app/flowix-desktop/src/providers/openai_compatible.rs`

当前状态：
- 已完成：请求总时长约束、429/5xx 初始重试。
- 未完成：SSE 仍使用 `String::from_utf8_lossy(&bytes)` 后按 `\n` 切行。

问题：
- 跨 chunk 的多字节 UTF-8 字符可能被替换为 `U+FFFD`。
- `\r\n`、`event:` 行、未来 provider 的 SSE 变体兼容性不足。

建议：
- 使用 `Vec<u8>` 累积 bytes。
- 找到完整行后再 UTF-8 decode。
- 支持 `\n` / `\r\n`，忽略或处理 `event:` 行。

### P1-2. 错误上下文仍然不足

位置：
- 多数 `commands/*.rs`
- `app/flowix-desktop/src/user_config.rs`
- `app/flowix-core/src/memo_file/*`

问题：
- 很多 IPC 返回 `bool` / `Option`，错误被吞掉或只 `eprintln!`。
- 前端只能知道失败，无法知道失败操作、路径、底层原因。

建议：
- 文件、导出、memo 写入类 IPC 改为 `Result<T, String>`。
- 错误消息包含操作名、路径、底层 IO/JSON/SQLite 错误。
- 前端 toast 展示用户可理解的错误。

### P1-3. 前端 ErrorBoundary 粒度和上报不足

位置：
- `app/flowix-web/App.tsx`
- `app/flowix-web/components/error-boundary.tsx`

问题：
- ErrorBoundary 主要包在根层，子树崩溃会让整个窗口进入 fallback。
- 仅 `console.error`，没有上报到后端日志或错误收集。

建议：
- 在 agent panel、memo pane、document editor、preferences window 增加局部 ErrorBoundary。
- 增加 `app-error` IPC，把前端错误写入后端日志文件。
- fallback 保留局部恢复按钮，不要让整窗不可用。

### P1-4. 可观测性仍不够

位置：
- `app/flowix-desktop/src/lib.rs`
- `app/flowix-desktop/src/main.rs`

当前状态：
- 已完成：tracing 初始化提前到启动开头。
- 未完成：日志仍主要输出到 stderr/stdout。

问题：
- Windows release 使用 `windows_subsystem = "windows"`，stderr 不可见。
- 用户无法提供可诊断日志。
- 没有 panic hook、日志轮转、PII 脱敏策略。

建议：
- 增加落盘日志，例如 `%APPDATA%/flowix/logs/flowix.log`。
- 配置日志轮转与最大大小。
- panic hook 写入日志，并在下次启动提示“上次异常退出”。

### P1-5. 搜索索引重建仍是粗粒度后台线程

位置：
- `app/flowix-desktop/src/commands/helpers.rs`

问题：
- `rebuild_index_in_background` 使用裸 `std::thread::spawn`。
- 重建期间持有 memo 读锁和 search 写锁，较大 notebook 下仍可能影响交互。
- 缺少取消、去抖、任务合并。

建议：
- 改为受控 worker/task。
- notebook 切换或连续变更时取消旧任务。
- 分批读取与增量 upsert，减少长时间持锁。

## P2 / 中优先级

### P2-1. CI/CD 缺失

位置：
- 仓库根目录缺少 `.github/workflows/`

问题：
- README 提到发布流程，但当前没有可执行 CI。
- PR、跨平台构建、release、审计都不可自动验证。

建议：
- 增加 PR workflow：`npm run build`、`cargo check`、`cargo test`。
- 增加 release matrix：Windows/macOS/Linux 分平台构建。
- 接入 `cargo audit` / `cargo deny` / `npm audit`。

### P2-2. Release profile 缺失

位置：
- `app/Cargo.toml`

问题：
- 缺少 `[profile.release]` 优化配置。
- 包体和冷启动可能偏大。

建议：
- 增加 `lto = "fat"`、`codegen-units = 1`、`strip = "symbols"`。
- 根据构建耗时权衡 `panic = "abort"`。

### P2-3. 打包目标与更新能力不足

位置：
- `app/flowix-desktop/tauri.conf.json`

问题：
- `bundle.targets: "all"` 不适合跨平台 CI matrix。
- 没有应用内 updater。

建议：
- 按 host 平台拆分 bundle target。
- 接入 `tauri-plugin-updater`。
- 配置签名、notarization、Windows signtool。

### P2-4. 前端测试体系缺失

位置：
- `package.json`
- `app/flowix-web/**`

问题：
- 前端没有 Vitest/Jest/RTL 测试配置。
- 关键 UI 状态、IPC adapter、store 行为缺少回归保护。

建议：
- 增加 Vitest + React Testing Library。
- 覆盖：settings 保存、agent runtime 切换、memo list filter/search、editor save CAS 失败提示。
- 增加一组 Playwright smoke test 覆盖主窗口启动和核心路径。

### P2-5. 依赖与 bundle 维护

位置：
- `package.json`
- `app/flowix-desktop/Cargo.toml`

问题：
- `rllm` 仍是关键 LLM 依赖，但维护状态需重新评估。
- `react-syntax-highlighter` 等大包如果仍进入主 chunk，会影响首屏。
- 需要定期依赖更新策略。

建议：
- 评估替换为 `reqwest + eventsource-stream` 自维护 provider。
- 对大体积前端依赖做 lazy import。
- 增加 Renovate/Dependabot。

## 已处理项

- P1-1 锁中毒恢复：已新增统一 lock helper 并替换生产路径关键 `RwLock::unwrap()`。
- P1-2 启动期初始化 panic：已改为 tracing 记录并降级到 transient/in-memory。
- P1-3 LLM 请求超时与重试：已增加默认 wall-clock timeout 和 429/5xx 初始请求重试。
